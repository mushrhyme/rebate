## 웹앱 워크플로우

### 역할 정의

| 역할                 | 대상                   | 권한                                               |
| -------------------- | ---------------------- | -------------------------------------------------- |
| **영업추진부** | 시스템 관리·전체 감독 | 모든 소매처 조회·편집, 매핑 확정, 1차 검토        |
| **영업사원**   | 담당 소매처 검토       | 본인 담당 소매처만 조회·편집, 매핑 확정, 2차 검토 |

> **담당 소매처 배정 전제조건**: 소매처 매핑이 확정되어 소매처코드가 결정된 이후에만 `retail_user.csv`를 통해 담당 영업사원이 배정된다. 매핑이 미확정인 소매처는 담당자를 알 수 없다.

---

### 전체 흐름

```
[영업추진부 또는 영업사원] PDF 업로드
      ↓
[시스템] Phase 1 — OCR → page MD 생성
[시스템] Phase 2 — 양식 식별 + 항목 추출
[시스템] Phase 3 — 소매처·제품코드 매핑 (캐시 히트 → 자동, 미확정 → 보류)
      ↓ 미확정 매핑 항목 있을 때
[영업추진부 또는 영업사원] 매핑 확정 (MappingReview 화면)
      — OCR 원문 확인 → 후보 목록에서 선택 또는 직접 검색 → 확정
      — 확정된 매핑은 캐시에 자동 저장 (다음 분석부터 자동 처리)
      ↓ 전체 매핑 확정 완료 시 자동 진행
[시스템] Phase 4 — NET 계산·교차검증 (자동 실행)
      ↓
[영업추진부] 1차 검토 (Results 화면)
      — 전체 소매처 대상
      — OCR 정확성 확인: OCR 원문 ↔ 청구서 원본 대조
      — 매핑 정확성 확인: OCR명 → 코드·매핑명 쌍 확인
      — 분류 정확성 확인: 타입 배지 (条件/CF/기타) 확인
      — NET 미달 확인: 빨간 ↓ 표시 항목 확인
      ↓
[영업사원] 2차 검토 (Results 화면)
      — 본인 담당 소매처만 표시 (기본값)
      — 필요 시 타인 담당 소매처 조회 가능 (편집 불가)
      — 1차 검토 결과 확인 + 이상 없으면 최종 확인
      ↓ 문서 내 모든 소매처의 1차+2차 완료 시 자동 진행
[시스템] 문서 확정 — 확정 타임스탬프 기록, 체크·매핑 편집 잠금
      ↓
[영업추진부] SAP 업로드 대상 선택 (SAP 업로드 화면)
      — 연월 / 문서명 / 확정여부 필터로 대상 문서 선택
      — 엑셀 미리보기 → 다운로드
```

### 한 소매처가 여러 페이지에 걸쳐 있는 경우

청구서 한 장에서 동일 소매처의 항목이 여러 페이지에 분산될 수 있다.

- **Results 화면**: 페이지와 무관하게 소매처 단위로 묶어서 표시한다.
- **PDF 뷰어**: 각 항목 행에 출처 페이지 번호가 표시된다. 클릭하면 PDF가 해당 페이지로 이동한다.
- **검토 단위**: 소매처 단위로 1차 완료 처리한다 (개별 행 단위가 아님).

---

### 문서 확정 상태

문서 내 모든 소매처의 1차+2차 검토가 완료되면 시스템이 자동으로 해당 문서를 **확정** 상태로 전환한다.

**확정 시 잠기는 것**

| 대상               | 잠금 내용                          |
| ------------------ | ---------------------------------- |
| 1차/2차 체크 버튼  | 변경 불가                          |
| 매핑 편집          | 소매처·제품 매핑 수정 불가         |

**확정 해제**

- 주체: 영업추진부 전용
- 위치: 대시보드에서 해당 문서의 확정 해제 실행
- 동작: 체크 상태는 그대로 유지하고 잠금만 해제 → 수정 후 재검토 완료 시 자동 재확정

### 경고 표시 기준

| 아이콘          | 의미                                          | 검토 시 확인 내용                                                       |
| --------------- | --------------------------------------------- | ----------------------------------------------------------------------- |
| `?` (노란 원) | 매핑 미확정 — 소매처 또는 제품 코드가 불확실 | 매핑 확정 단계에서 처리됐어야 하는 항목. Results에 남아있으면 이상 상태 |
| `↓` (빨간)   | NET이 본부장 기준 미달                        | 영업사원에게 경각심 전달 목적. 수치 자체는 계산 오류가 아님             |

### 미구현 예정 기능 (설계 확정, 미작성)

> 구현 완료된 항목은 이 목록에서 제거한다.

- 영업사원 담당 소매처 필터 자동 적용 — API(`GET /api/v3/retailers/my`)는 구현됨. Results 화면에서 영업사원 로그인 시 기본값으로 본인 담당만 표시되도록 UI 자동 적용 미완
- Results 소매처 단위 이전/다음 네비게이션
- 항목 행 클릭 시 PDF 출처 페이지 이동
- 문서 확정 플래그 — DB 컬럼(`v3_documents.confirmed_at`) 추가 + 모든 소매처 1차+2차 완료 시 자동 세팅 API
- 확정 해제 API (영업추진부 전용, 대시보드에서 실행)
- 확정 문서 체크·매핑 편집 잠금 UI
- SAP 업로드 화면 — 연월·문서명·확정여부 필터, 엑셀 미리보기·다운로드
- 대시보드 확정 상태 표시 (`v3_documents.status` 값: `ocr` → `analyzing` → `pending` → `done` → **`confirmed`** 추가 예정)

## 분석 사이클

```
PDF 파일
  └─ (OCR이 안 되어 있으면 자동 실행)
      ↓
Phase 1 — page MD 생성
  └─ 검증 결과 이상 시 ⚠️ 확인 요청 ← 중단점 ①
      ↓
Phase 2 — 양식 식별 + 항목 추출
  └─ 처음 보는 양식이면 form 정의 작성 ← 중단점 ②
      ↓
Phase 3 — 소매처·제품코드 매핑
  └─ 매칭 불확실 항목마다 ← 중단점 ③
      ↓
Phase 4 — NET 계산·교차검증 (Python 자동 실행)
      ↓
결과 표시
```

**중단점에서 Claude가 멈추는 경우:**

① Phase 1 검증 — MD 생성 품질에 중간 이상 문제가 있을 때. 수정 후 진행 또는 그대로 진행 중 선택.

② 신규 양식 — 등록되지 않은 양식이 들어왔을 때. Claude와 함께 양식 정의를 작성한 뒤 승인하면 자동으로 이어진다.

③ 매핑 확인 — 소매처명 또는 제품명이 불확실할 때. 후보 목록을 보여주고 번호를 선택하거나 직접 코드를 입력한다. 확정된 매핑은 자동으로 캐시에 저장되어 다음번에는 묻지 않는다.

## 명령어 상세

---

### 청구서 분석 (`analyze-invoice`)

**대상: 현업**

**이렇게 말하면 됩니다**

```
sample_003 분석해줘
三菱食品東日本_2025.01 분석해줘
이 청구서 분석해줘
```

**하는 일**

Phase 1(page MD 생성) → Phase 2(항목 추출) → Phase 3(소매처·제품 매핑) → Phase 4(NET 계산) 전체를 순서대로 실행한다.

- OCR 파일이 없으면 자동으로 Azure OCR을 먼저 실행한다.
- 이미 생성된 page MD가 있으면 재생성 없이 기존 파일을 사용한다.
- Phase 4 결과는 `extracted/{doc_id}/phase4_output.json`에 저장된다.

**파일 위치 규칙**

| 입력                               | Claude가 찾는 경로                     |
| ---------------------------------- | -------------------------------------- |
| `sample_003 분석해줘`            | `samples/03/sample_003.pdf`          |
| `samples/02/sample_002 분석해줘` | `samples/02/sample_002.pdf` (그대로) |

PDF 파일이 없으면 안내 후 중단한다.

---

### PDF 텍스트 추출 (`azure-ocr`)

**대상: 현업**

**이렇게 말하면 됩니다**

```
sample_003 OCR해줘
PDF 텍스트 뽑아줘
이 PDF OCR해줘
```

**하는 일**

Azure Document Intelligence로 PDF를 분석해 페이지별 텍스트 파일을 생성한다.

출력 위치: `samples/{NN}/{doc_id}_pages/page_NNN.ocr.txt`

**사전 조건**: `.env`에 `AZURE_API_KEY`와 `AZURE_API_ENDPOINT`가 설정되어 있어야 한다. 없으면 안내 후 중단.

> 청구서 분석(`분석해줘`) 실행 시 OCR 파일이 없으면 자동으로 이 단계가 먼저 실행된다. 별도로 호출할 필요는 없다.

---

### Form 정의 업데이트 (`update-form`)

**대상: 현업**

**이렇게 말하면 됩니다**

```
(이미지 첨부) form_04에 반영해줘
(이미지 첨부) 업데이트해줘
(이미지 첨부) 추가해줘
```

**이미지를 반드시 첨부해야 한다.** 이미지 없이 "반영해줘"만 입력하면 트리거되지 않는다.

**하는 일**

첨부한 이미지를 읽어 해당 양식의 `form_definitions/form_XX.md`를 업데이트한다.

이미지 유형에 따라 업데이트 대상 섹션이 달라진다:

| 이미지 유형                   | 업데이트 대상                           |
| ----------------------------- | --------------------------------------- |
| 실제 청구서 화면 캡처         | 추출 컬럼, OCR 정규화 규칙, 함정 케이스 |
| 업무규칙 자료 (PPT·Excel 등) | タイプ 분류 규칙, NET 계산식            |
| 양식 구조 설명 자료           | 문서 구조, 페이지 역할                  |

변경 내용을 보여주고 승인을 받은 후에만 저장한다.

---

### Form 설정 동기화 (`sync-form-config`)

**대상: 현업**

**이렇게 말하면 됩니다**

```
form_04 수정했어, 동기화해줘
md 바꿨는데 반영해줘
업무규칙 변경했어
동기화해줘
```

**하는 일**

`form_definitions/form_XX.md`를 읽어 `config/form_types.json`을 자동으로 업데이트한다.

- form_XX.md의 `[Phase 3] タイプ 분류 규칙`에서 `✅ 확정` 항목만 반영한다.
- form_XX.md의 `[Phase 4] NET 계산식`에서 수식 타입과 파라미터를 반영한다.
- 변경 내역을 보고한다. 개발자 확인이 필요한 케이스는 ⚠️ 표시.

**form_types.json을 직접 편집하지 않는다.** form_XX.md를 수정한 뒤 이 명령을 실행한다.

### 신규 양식 등록

**대상: 현업 (자연어) + 개발자·운영자 (/cold-start)**

**현업 — 이렇게 말하면 됩니다**

```
새 양식 등록해줘
처음 보는 청구서야, 양식 만들어줘
```

**개발자 — 이렇게 입력합니다**

```
/cold-start
/cold-start sample_007
```

**하는 일**

처음 보는 양식의 `form_definitions/form_XX.md`를 Claude와 대화로 작성한다.

1. 현재 등록된 양식과 다음 form_id 확인
2. doc_id를 전달하면 해당 문서의 page MD를 읽어 양식 구조 파악
3. 발행처명, 컬럼 구성, 업무규칙 순서로 확인
4. 초안 작성 후 승인 요청 → 승인 시 저장 + `_index.md` 등록

**주의:** 대화에서 발행처명·컬럼 구성·업무규칙은 현업 도메인 지식이 필요하다. **반드시 현업이 대화에 참여해야 하며**, 개발자 단독으로 진행하지 않는다.

**사용자 승인 없이 파일을 저장하지 않는다.**

> 청구서 분석 중 처음 보는 양식을 만나면 자동으로 이 흐름으로 진입하며, 완료 후 분석이 이어서 진행된다. 현업은 `분석해줘`만 실행해도 된다.

---

### 타입 분류 규칙 확정 (`update-form`)

**대상: 현업**

**이렇게 말하면 됩니다**

```
form_04에서 이 규칙 확정해줘
CF 키워드에 宅配掲載料 추가해줘
업무규칙이 바뀌었어 — 備考가 *이면 10% 적용이 아니라 8%야
```

**하는 일**

cold-start 이후 `TBD`로 남겨진 타입 분류 규칙을 현업 확인 후 `✅ 확정`으로 업데이트한다.

- Claude가 현재 form_XX.md를 읽고 변경 내용을 정리해 보여준다.
- 승인하면 form_XX.md를 저장한다.
- 저장 후 form_types.json 동기화 여부를 물어본다.

> 이미지 없이 텍스트로만 요청해도 된다. 이미지가 있으면 같은 명령(`반영해줘`)을 쓰면 된다.

---

### Phase 4 재실행 (`/phase4`)

**대상: 개발자·운영자**

**이렇게 입력합니다**

```
/phase4 sample_003
/phase4 三菱食品東日本_2025.01
```

**하는 일**

Phase 3까지 완료된 문서에 대해 Phase 4(NET 계산·교차검증)만 단독으로 다시 실행한다.

```
python3 scripts/phase4_calc.py --doc {doc_id} --save
```

- `extracted/{doc_id}/phase3_output.json`이 없으면 안내 후 중단.
- 결과는 `extracted/{doc_id}/phase4_output.json`에 저장.

> 매핑 코드나 계산 로직을 수정한 뒤 계산만 다시 돌릴 때 사용한다.

## 파일 구조 (현업 소유)

현업이 관리하는 영역은 아래 두 곳이다. **파일을 직접 편집하지 않는다.** 내용은 Claude와 대화로 만들고, 승인하면 Claude가 저장한다.

```
form_definitions/
  ├── _index.md          # 등록된 양식 목록
  ├── form_01.md         # FINET 양식 정의
  ├── form_04.md         # 日本アクセス CVS 양식 정의
  └── image/             # 근거 이미지 보관 (update-form 실행 후 저장)

mappings/
  ├── ocr_retailer.csv   # 소매처 매핑 캐시 (분석 시 자동 누적)
  └── ocr_product.csv    # 제품 매핑 캐시 (분석 시 자동 누적)
```

`.claude/`, `scripts/`, `docs/`, `config/` 는 개발자 소유 — 편집하지 않는다.
