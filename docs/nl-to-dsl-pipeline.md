# 자연어 → DSL 컴파일 파이프라인 (LLM 산식 작성, 결정적 실행)

**작성일**: 2026-06-17
**대상**: 개발자 — 설계 합의용. 현업이 자연어로 계산·표시 규칙을 만들고, 결정적 엔진이 실행하는 구조.
**관련**: [phase4-dsl-readiness.md](phase4-dsl-readiness.md)(기존 DSL 백본), [architecture.md](architecture.md) §3·§7·§10

---

## 1. 배경 — 왜 이 문서가 필요한가

현재 원칙(architecture.md §3): **"회계 산수는 LLM이 하지 않는다. 분류·식별은 Claude, 계산은 결정적 코드. 재현성이 생명."**

이 원칙 때문에, 이번 `product_aggregate`(제품별 이중조건 수량 분해)처럼 **새 계산·표시 로직**은 현업이 자연어로 요청해도 반영되지 않고 개발자 코드 작업이 필요했다. 현업은 이 경계를 자연어로 넘고 싶어 한다.

**그러나 원칙을 폐기하면 안 된다.** 이번 "왜 60이 안 나오나" 디버깅이 그 증거다 — 숫자가 결정적이라서 API 응답을 직접 호출하고, 골든 테스트를 돌리고, 재현해서 끝내 원인(프론트 키의 널바이트)을 잡았다. 만약 LLM이 분석 때마다 숫자를 즉석 계산했다면 "맞는 값"의 기준 자체가 없어 추적 불가였다.

**해법: 원칙을 폐기가 아니라 이동한다.**

---

## 2. 재정의된 원칙

> **LLM은 산식을 *작성*하되 *실행*하지 않는다.**

```
자연어(현업) → [LLM 컴파일] → 검증가능한 DSL 스펙 → [샌드박스 검증·승인] → 동결 → [결정적 엔진 실행]
                                                                                      └ 런타임 숫자는 여전히 코드가 만든다
```

바뀌는 것: **"누가 산식을 *만드느냐"*** (개발자 → LLM+현업).
불변인 것: **"누가 숫자를 *계산하느냐"*** (영원히 결정적 코드).

LLM이 SAP로 나갈 리베이트 숫자를 직접 계산하는 일은 없다. LLM은 그 숫자를 만드는 *산식*을 자연어에서 번역하고, 그 산식은 검증·동결되기 전까지 실행되지 않는다.

---

## 3. 현황 — 백본은 이미 있다

phase4-dsl-readiness.md 기준, **결정적 실행·검증 인프라가 상당 부분 구현돼 있다.** 이것이 이 설계의 출발점이자 핵심 de-risk 요인이다.

| 구성요소 | 상태 | 위치 |
|---|---|---|
| `_safe_eval` AST 기반 안전 평가기 (eval() 미사용, 화이트리스트) | ✅ | scripts/phase4_calc.py |
| `_eval_expr` expr 경로 (vars / computed_vars / divide_by) | ✅ | scripts/phase4_calc.py |
| `FORMULA_REGISTRY` plugin 경로 | ✅ | scripts/phase4_calc.py |
| JSON Schema 검증 (`form_types.schema.json`) | ✅ | config/ |
| 회귀/골든 테스트 (fixtures 기반) | ✅ | tests/regression/, tests/fixtures/ |

즉 "안전한 수식 평가기 + 스키마 + 골든 회귀"라는 **결정적 실행기와 검증 도구는 존재한다.** 이 설계는 그 위에 두 가지를 얹는 것이다.

## 4. 갭 — 무엇이 없는가

| 갭 | 내용 | 난이도 |
|---|---|---|
| **G1. NL→DSL 컴파일을 채팅 1차 경로로** | 지금은 sync-form-config가 form_XX.md에서 net.expr을 생성. 현업 자연어 → DSL을 대화의 기본 흐름으로 만들고, 검증 결과를 사용자에게 노출 | 중 |
| **G2. 사용자향 검증 게이트** | 지금 검증은 CLI/개발자측(pytest·schema). 이걸 "샘플 숫자 + 골든 diff를 사람이 승인"하는 UI/대화 게이트로 surface | 중 |
| **G3. DSL 어휘 확장(연산)** | 기존 expr은 **행 단위 NET 수식**만 표현. `product_aggregate` 같은 **그룹 단위 집계 변형**은 새 인터프리터 코드가 필요했음 | 높 |
| **G4. 화면(표시) 레이어** | `product_aggregate` 표시는 Results.tsx 수작업 React. 표시 형태 변경은 아직 프론트 코드 필요 | 높(가장 어려움) |

**핵심 통찰:** G3·G4가 진짜 경계다. 행 단위 수식은 이미 DSL. 새로운 *연산 종류*(집계 변형·표시 형태)는 (a) plugin/인터프리터 코드(개발) 또는 (b) 더 일반적인 안전 변환 DSL이 필요하다. 무리하게 (b)로 임의 표현을 허용하면 위험하므로, **연산은 화이트리스트로만 확장**한다.

## 5. 목표 파이프라인

현업 제안을 기존 백본에 맞춰 구체화:

| 단계 | 내용 | 재사용/신규 |
|---|---|---|
| ① 자연어 의도 | 현업이 채팅으로 "이 양식은 定番에서 추가조건 빼서 분해" 표현 | — |
| ② LLM 컴파일 | 자연어 → **구조화 DSL**(form_types.json 형식, 자유 산문 ❌) | 신규(G1) |
| ③ 스키마 검증 | 형식·필수필드·타입·**금지문자(널바이트 등)** | 기존 schema + 정규화 보강 |
| ④ 샌드박스 dry-run | 샘플 문서에 결정적 실행 → **실제 숫자 산출** | 기존 phase4_calc |
| ⑤ 불변식 + 골든 diff | 합계 보존("분해 후 총금액=원본") 등 + 기존 확정 결과와 차이 표 | 기존 regression + 불변식 신규 |
| ⑥ 사람 승인 → **동결** | 컴파일된 DSL을 버전 고정(자연어 아님), 변경 이력 | 신규(G2) |
| ⑦ 재분석 → 확인 | 동결 DSL을 결정적 실행, 화면 확인 | 기존 |

**동결 원칙:** source of truth는 *컴파일된 DSL*이지 *자연어*가 아니다. 자연어를 매번 LLM에 재통과시키면 결과가 드리프트한다. 자연어는 변경 의도의 기록으로만 남긴다.

## 6. 검증 게이트 스펙 (③④⑤가 시스템의 심장)

각 게이트는 **통과 못 하면 ⑥ 승인 화면에 도달 못 함**:

1. **스키마**: form_types.schema.json 통과. + **생성물 정규화**(제어문자·널바이트 거부, 공백 정규화). ← 이번 버그의 직접 교훈.
2. **dry-run**: 지정 샘플 doc_id들에 phase4 실행. 예외 없이 완주.
3. **불변식**(연산별 정의):
   - 분해류: Σ분해수량·금액 = 원본 제품 총량/총금액 (±0.01)
   - NET류: 기존 fixture와 행수·값 일치(의도된 변경이면 diff 승인)
   - 음수/0나누기: 경고로 표면화, 침묵 금지
4. **골든 diff**: 기존 확정 결과 대비 바뀐 셀을 표로 제시. 의도된 변경만 사람이 승인.
5. **표시 dry-run**(G4 해당 시): 표시 스펙을 실제 렌더 데이터에 적용해 키 매칭률 100% 확인. ← 이번 aggMap 미스가 여기서 걸렸어야 했다.

## 7. DSL 어휘 — 3티어로 관리

| 티어 | 정의 | 반영 경로 | 위험 |
|---|---|---|---|
| **T1. 기존 어휘의 새 인스턴스** | 다른 양식에 동일 패턴 적용(예: 이중조건 분해 재사용, 새 net.expr) | **자연어 → DSL 설정, 검증 게이트만으로 완결** | 낮음 — 지금도 거의 가능 |
| **T2. 기존 연산의 파라미터 확장** | 분해 기준 조건 변경, 컬럼 구성 변경 등 | DSL 설정 + 게이트 | 낮~중 |
| **T3. 완전히 새로운 연산 종류** | DSL에 없는 집계·표시 유형 | **인터프리터/plugin 코드(개발) 또는 화이트리스트 연산 추가** + 게이트 | 높음 |

**운영 규칙:** 새 *유형*(T3)은 개발이 화이트리스트에 연산을 추가하는 1회 작업. 그 이후 같은 유형의 *적용*(T1/T2)은 자연어+게이트로 내려온다. `product_aggregate`가 정확히 이 경로를 밟았다(코드 1회 → 이후 설정 재사용). **DSL은 임의 코드 실행을 절대 허용하지 않는다 — 연산은 등록된 화이트리스트로만.**

---

## 8. 화면(표시) 레이어 — 가장 어려운 갭(G4)

행 데이터 DSL과 달리, 표시(컬럼·분해 행·표 형태)는 현재 Results.tsx 수작업이다. 단계적 접근:

- **8a (현실적 1차):** 표시 스펙을 **선언적 설정**으로(예: `condition_columns`, 행 종류). 프론트는 그 설정을 해석하는 **범용 렌더러** 1개를 갖는다. 새 표시 형태가 이 렌더러의 표현 범위 안이면 설정만으로 가능.
- **8b (확장):** 렌더러가 표현 못 하는 형태는 프론트 코드(개발). T3와 동일 구조.
- **불변:** 표시 로직은 숫자를 만들지 않는다. 백엔드가 만든 값을 배치만 한다(architecture.md 원칙 유지).

## 9. 단계별 로드맵

| Phase | 산출물 | 상태 |
|---|---|---|
| **P0 (이 문서)** | 설계 합의 + 의사결정 기록(architecture.md §7) | ✅ 완료 |
| **P1. 검증 게이트** | `scripts/validate_dsl_gate.py` — 스키마+제어문자 / dry-run+필드실재성 / 불변식 / 골든 diff. 네거티브로 오작성·널바이트 차단 확인 | ✅ 완료 |
| **P2. NL→DSL 컴파일 PoC** | `scripts/compile_dsl_poc.py` — 자연어→Claude(Sonnet 4.6 tool_use)→설정→P1 게이트 통과 | ✅ 완료 |
| **P3. 승인 게이트 + 동결** | `scripts/dsl_apply.py` — 미리보기(승인 요약)→승인→백업·form_types.json 동결·변경이력·사후재검증. 게이트 비검증 필드는 ⚠+`--confirm-display` 가드 | ✅ 완료 (CLI) |
| **P4. 표시 렌더러(8a)** | `frontend/src/components/aggTable.tsx` — 컬럼 스펙(kind: mark/qty/unit/amount) + 범용 셀 렌더러. Results.tsx 제품별 집계 표가 인라인 JSX 대신 스펙 사용 | ✅ 완료 (1차) |

**P4 완전판 ✅:** 백엔드 `build_product_aggregate`가 `display_columns`(kind 기반 스펙) emit → 프론트는 그걸 우선 해석(없으면 condition_columns 폴백). '새 표 형태 = 스펙 변경, 프론트 코드 0' 성립.
**P4 남은 것(8b):** 그룹 헤더 행(제품명+소계)까지 스펙화하면 표 구조 전체가 선언적. 표시 스펙을 form_types.json에 두고 양식별로 다르게 emit하는 것도 가능.

**P1~P3에서 배운 것:** ① 게이트 자체도 검증해야 한다(필드 실재성 누락을 네거티브 테스트로 발견). ② 게이트는 *숫자에 영향 주는 필드*만 검증 가능 — 표시 전용 필드(unit_field)는 **사람 승인 단계가 잡아야 함**(LLM의 unit_field 환각이 게이트는 통과하되 승인 요약에서 ⚠로 표면화됨). ③ 동결 대상은 컴파일된 설정이지 자연어가 아니다.
**P3 UI 연동 ✅:** 백엔드 라우트 `POST /api/v3/dsl/preview`(누구나)·`/apply`(관리자) + 프론트 `pages/DslStudio.tsx`(관리자 네비 '규칙 스튜디오'). 자연어 입력→검증(컴파일+게이트+diff+표본)→승인·동결. CLI(`scripts/dsl_apply.py`)는 개발/배치용으로 병존.
NOTE: 컴파일/게이트는 `scripts/`를 런타임 import(PoC) — 운영 승격 시 `backend/`로 이동 권장.

## 10. 리스크·가드레일 (비협상)

| 리스크 | 가드레일 |
|---|---|
| 생성물 오염(널바이트·제어문자) | 게이트①에서 정규화/거부. **이번 버그의 직접 교훈.** |
| 그럴듯하게 틀린 산식 | 게이트③④ — LLM이 썼다 ≠ 검증됐다. **실제 숫자를 돌려 사람이 확인**이 유일한 안전망 |
| 재현성/드리프트 | 컴파일된 DSL을 동결. 자연어 재실행 금지. source of truth = 컴파일 산출물 |
| DSL 범위 폭주 | 연산은 화이트리스트 + 제한된 표현식. `_safe_eval` 원칙 유지. 임의 실행 영구 금지 |
| 표시 미스(키 불일치 등) | 게이트⑤ 표시 dry-run에서 매칭률 100% 강제 |
