# Cold-start — 신규 양식 자동 대응

> 이 분기는 **이 리팩의 존재 이유**다. 곁가지가 아니라 메인 파이프라인의 일급 분기로 다룬다.

> **⚠️ 이 문서는 Phase B (React UI) 설계서다.**  
> Phase A (현재 운영 중인 Claude Code 채팅 방식)의 신규 양식 대응 흐름은 [docs/framework.md](framework.md) — "마중물 단계" 섹션을 참조할 것.

## 1. 왜 1급 분기인가

기존 시스템에서 신규 양식이 들어오면:

1. RAG가 *유사 페이지 0건* → 엉뚱한 양식 예시를 가져옴
2. LLM이 의미 없는 추출 결과 반환
3. 개발자가 수동으로 정답지(`is_answer_key_document`)를 만들어야 비로소 학습 시작
4. 그동안 사용자는 분석 결과를 받지 못함

새 시스템은 사용자 PDF 업로드 시점에:

1. 신규 양식임을 자동 감지 (양식 식별기 confidence < 임계)
2. 기존 양식 정의들을 *형식 예시*로 보고 신규 양식 정의 초안 자동 생성
3. 같은 호출에서 시험 추출까지 수행
4. 사용자에게 추출 결과 + 정의 초안을 함께 보여줌
5. 사용자가 검토·수정·승인 → 코드 배포 없이 즉시 반영, 다음 동일 양식부터 자동

## 2. 트리거 조건

다음 중 하나라도 충족하면 cold-start 분기 진입:

- 양식 식별기(Haiku)가 `confidence < 0.7` 반환
- 사용자가 업로드 시 "신규 양식" 체크박스로 명시적 지정
- 첫 시험 추출의 항목별 confidence 평균이 임계 미만

오탐(기존 양식인데 신규로 잘못 분류)은 사용자 화면에서 "기존 양식 X로 처리" 옵션으로 우회 가능.

## 3. 자동 form_NEW.md 생성

### 입력
- 이 PDF의 모든 page MD ([architecture.md](architecture.md) Phase 1 결과)
- `form_definitions/*.md` 전체 (기존 양식 정의들 — *형식 예시 풀*)
- `form_definitions/_index.md` (양식별 1줄 요약)

### Claude 호출 (Opus 권장 — 1회성, 어려움)

프롬프트 골자:

```
당신은 청구서 양식 정의서를 작성합니다.
아래는 이미 정의된 양식들입니다 — 이 *형식*을 따라 새 양식 정의를 작성하세요:

<example>{form_01.md 전체}</example>
<example>{form_02.md 전체}</example>
... (모든 기존 양식)

다음은 새로 들어온 양식의 페이지 MD입니다:
{page_001.md … page_NNN.md}

작업:
1. form_NEW.md 초안 작성
   - 식별 단서, 페이지 역할, 필드 정의, 분류 어휘, 함정 케이스
2. form_types.json에 추가할 항목 후보 작성
   - fields, calc_rules, sap_quantity, sap_extra_columns
3. 같은 호출에서 시험 추출 수행
   - 위 정의 초안을 적용했다고 가정하고 items[] 생성
4. 불확실한 항목은 ⚠️ 표시
```

### 출력
- `form_definitions/_drafts/form_NEW_<timestamp>.md` — 정의서 초안
- `extracted/<doc_id>/_coldstart.json` — 시험 추출 결과 + 룰 후보 + 신뢰도 메타

## 4. 사용자 검토 화면

### 레이아웃 (안)

```
┌────────────────────────────┬──────────────────────────────────┐
│  [좌] 시험 추출 결과 그리드   │  [우] 자동 생성 정의서             │
│                            │                                  │
│  - 행마다 confidence 표시   │  ┌─[탭1] form_NEW.md ──────────┐ │
│  - 행 클릭 시 source_page   │  │  (마크다운 편집기)            │ │
│    원본 이미지 점프          │  │                              │ │
│  - flagged 행은 빨간 배경    │  └────────────────────────────┘ │
│                            │  ┌─[탭2] 계산식 (form_types.json) │
│  [PDF 페이지 미리보기]      │  │  - fields                    │ │
│                            │  │  - calc_rules                │ │
│                            │  │  - sap_quantity              │ │
│                            │  │  - sap_extra_columns         │ │
│                            │  └────────────────────────────┘ │
│                            │                                  │
│                            │  [재시험 추출]   [승인 저장]       │
└────────────────────────────┴──────────────────────────────────┘
```

### 사용자 동작 흐름

1. 그리드 결과를 보고 잘못 추출된 행 식별
2. 우측 정의서를 수정 (예: "이 양식은 단가가 표지가 아니라 page=2에 있다" 추가)
3. **재시험 추출** 버튼 → 수정된 정의로 같은 PDF 재추출 → 그리드 갱신
4. 결과가 만족스러우면 **승인 저장**:
   - `form_definitions/form_NEW.md` 정식 위치로 이동 (drafts 폴더 빠져나감)
   - `config/form_types.json`에 항목 추가
   - `form_definitions/_index.md`에 한 줄 요약 추가
   - 이 PDF의 items도 확정 → DB 저장
5. 다음 동일 양식 PDF 들어오면 → 정상 분기로 자동 처리

## 5. UI 옵션 (단계적 발전)

### MVP (먼저 구현)
- raw MD 편집기 (Monaco) + 계산식 JSON 편집기
- 개발자/파워유저용 — 빠르게 만들 수 있고 유연함

### 안정화 후
- **폼 위에 폼**: MD를 파싱해서 "필드 목록", "페이지 역할", "분류 어휘", "계산식" 섹션으로 분리한 폼 제공
- 저장 시 다시 MD/JSON으로 직렬화
- 회계·총무 일반 사용자 대상

### 하이브리드 (선택)
- 기본은 폼, "고급 사용자" 토글로 raw MD 모드 제공

## 6. 사용자 부담 최소화 원칙

- **사용자는 코드를 짜지 않는다.** 자연어 MD + JSON 룰만 다룬다.
- **자동 생성이 80%를 해결한다.** 사용자는 검수·미세 조정만.
- **언제든 되돌릴 수 있다.** 정의서는 git/DB로 버전 관리. 잘못 학습됐으면 이전 버전으로 롤백.
- **재처리 가능.** 정의 수정 후 과거 PDF를 같은 양식으로 다시 돌릴 수 있음.

## 7. 운영 기준

- `form_definitions/_drafts/`에 7일 이상 머문 초안은 알림 (방치 방지)
- 한 양식의 정의가 7일 내 3회 이상 수정되면 "불안정 양식" 태깅 → 운영자 검토 권장
- 분기별로 cold-start 통과율(자동 생성 → 첫 승인까지 사용자 수정량) 메트릭화

## 8. 기존 양식 5개도 이 형식으로 작성

리팩 시 기존 form_01~05의 양식별 Python utils를 *수동으로* `form_definitions/form_01.md` … `form_05.md`로 옮긴다.

**이 5개가 cold-start의 few-shot 예시 풀이므로 일관된 형식으로 작성하는 것이 시스템 전체 품질의 토대다.**

각 정의서에 포함할 섹션:

- **식별 단서** — 표지 키워드, 로고, 레이아웃 특징
- **페이지 역할** — cover / detail / summary 별 위치와 단서
- **필드 정의** — 논리키 ↔ 물리 위치
- **분류 어휘** — `applied_conditions`로 쓸 토큰 목록과 의미
- **함정 케이스** — "이 양식은 X를 Y로 잘못 읽기 쉬움. Z를 확인해서 보정"
- **샘플 ID** — `samples/`에 있는 검증용 PDF 파일명

## 9. 평가 지표 (PoC에서 측정)

cold-start의 *합격 기준*은 다음 두 메트릭으로 판정:

| 지표 | 기대값 | 측정 방법 |
|------|--------|----------|
| **자동 생성된 정의서의 사용자 수정량** | 5줄 이하 | git diff 라인 수 |
| **첫 시험 추출 정확도** | 항목 정확도 ≥ 80% | 사람 검수 결과와 비교 |
| **승인까지 걸린 시간** | 30분 이하 | 화면 진입 → 승인 클릭 |

이 셋이 만족되면 *기존 시스템의 RAG 학습 루프를 완전히 대체할 수 있음*이 검증된 것이다.
