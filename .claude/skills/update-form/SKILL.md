# Form 정의 업데이트

## 입력 유형

이 스킬은 두 가지 경로로 진입한다.

| 경로 | 조건 | Step 1 처리 |
|------|------|------------|
| **이미지 경로** | 이미지 첨부 또는 image/ 경로 언급 | Step 1-A 실행 |
| **텍스트 경로** | 이미지 없이 규칙·키워드·업무규칙 변경 요청 | Step 1-B 실행 |

---

## 절차 (반드시 이 순서대로, 건너뛰지 않는다)

### Step 1-A — 이미지 읽기 (이미지 경로)

채팅 첨부 이미지: 그대로 읽는다.
파일 경로 언급: `Read` 도구로 해당 경로를 읽는다.

이미지 유형을 아래 셋 중 하나로 판정한다:

| 유형 코드 | 해당하는 경우 | 업데이트 대상 섹션 |
|----------|-------------|-----------------|
| `SAMPLE` | 실제 청구서·명세서 화면 캡처 | 추출 컬럼, OCR 정규화 규칙, 함정 케이스, 출력 예시 |
| `RULE`   | PPT·Excel·메모 등 업무규칙 기술 자료 | タイプ 분류 규칙, NET 계산식 |
| `SCHEMA` | 양식 구조·흐름 설명 자료 | 문서 구조, 문서 내 계층 구조, 페이지 역할 |

### Step 1-B — 텍스트 입력 파악 (텍스트 경로)

사용자 메시지에서 변경 요청을 직접 파악한다.

- 어떤 form_id인지 명시됐으면 그대로 사용. 아니면 Step 2에서 확인.
- 변경 내용 유형을 아래 중 하나로 분류한다:

| 요청 유형 | 예시 | 대상 섹션 |
|----------|------|----------|
| TBD → 확정 | "이 규칙 확정해줘" | [Phase 3] タイプ 분류 규칙 비고 |
| 키워드·항목 추가 | "CF 키워드에 X 추가해줘" | [Phase 3] タイプ 분류 규칙 |
| 업무규칙 변경 | "業務규칙이 바뀌었어 — ..." | [Phase 3] or [Phase 4] |
| 계산식 변경 | "NET 계산이 달라졌어 — ..." | [Phase 4] NET 계산식 |

파악된 변경 내용이 불명확하면 사용자에게 짧게 물어 명확히 한 뒤 Step 2로 진행한다.

---

### Step 2 — 대상 form 확인

사용자가 form_id를 명시한 경우: 그대로 사용한다.
명시하지 않은 경우: 이미지 내용(발행처명, 양식 특징)으로 추론 후 **반드시 사용자에게 확인한다.**

```
이미지를 보니 {발행처명} 양식으로 보입니다 (form_XX).
맞으면 계속하겠습니다.
```

form_id가 확정되면 `form_definitions/form_XX.md`를 읽는다.
파일이 없는 경우: `/cold-start` 흐름으로 전환한다.

### Step 3 — 변경 내용 도출

**이미지 경로:**
Step 1-A에서 판정한 유형의 대상 섹션을 순서대로 검토한다.
이미지에서 읽은 내용과 현재 form_XX.md를 비교해 추가·수정이 필요한 항목을 추출한다.
- 이미지에 있는 컬럼·규칙이 form_XX.md에 없으면 → 추가
- form_XX.md 내용과 이미지가 다르면 → 수정
- form_XX.md에 있고 이미지와 동일하면 → 변경 없음 (해당 섹션 생략)

**텍스트 경로:**
Step 1-B에서 파악한 요청 내용을 기준으로 변경 항목을 도출한다.
- 사용자가 명시한 규칙·키워드·수식만 변경. 언급하지 않은 항목은 건드리지 않는다.
- TBD → ✅ 확정 요청인 경우: 해당 규칙 행의 비고만 교체하고 규칙 내용은 그대로 유지.

### Step 4 — 업데이트 제안 표시 (고정 포맷)

**반드시 아래 포맷 그대로 출력한다. 포맷을 바꾸지 않는다.**

이미지 경로:
```
## form_XX.md 업데이트 제안

### 이미지 해석
유형: {SAMPLE | RULE | SCHEMA}
내용 요약: {한 줄}
대상 양식: form_XX — {발행처명}

---

### 변경 사항 ({N}건)

#### {번호}. {섹션명}

현재:
{현재 섹션 내용 — 없으면 "(없음)"}

변경 후:
{변경 후 섹션 내용}

---

저장하겠습니다. 수정할 내용이 있으면 말씀해 주세요.
```

텍스트 경로:
```
## form_XX.md 업데이트 제안

요청 내용: {사용자 요청 한 줄 요약}
대상 양식: form_XX — {발행처명}

---

### 변경 사항 ({N}건)

#### {번호}. {섹션명}

현재:
{현재 섹션 내용}

변경 후:
{변경 후 섹션 내용}

---

저장하겠습니다. 수정할 내용이 있으면 말씀해 주세요.
```

변경 사항이 0건인 경우:
```
## form_XX.md 업데이트 제안

검토했으나 form_XX.md에 반영할 변경 사항이 없습니다.
이미 동일한 내용이 정의되어 있거나, 요청에서 새로운 정보를 추출하지 못했습니다.
```

### Step 5 — 사용자 확인 대기

사용자가 수정 없이 진행을 승인하거나 "응", "저장해" 등을 입력하면 Step 6으로 진행한다.
수정 요청이 있으면 해당 항목만 재도출하고 Step 4 포맷으로 다시 표시한 뒤 재확인한다.

### Step 6 — 저장

`Edit` 도구로 `form_definitions/form_XX.md`의 해당 섹션만 수정한다.
전체 파일을 재작성하지 않는다. 변경 대상 섹션만 정확히 교체한다.

신규 양식인 경우: `form_definitions/_index.md`에 1줄 추가한다.

저장 완료 후 아래 형식으로 마무리한다:

> **Literate config**: 실행에 영향을 주는 규칙(NET 수식·タイプ·교차검증·출력설정 등)을 바꿨다면, form_XX.md의 `## [config]` **정본 블록**도 같은 변경으로 갱신해야 한다(정본은 블록, form_types.json은 생성물). 블록을 갱신했으면 `python scripts/build_form_types.py`로 재빌드한다. 설계: [docs/literate-config-migration.md](../../../docs/literate-config-migration.md)

이미지 경로인 경우:
```
form_XX.md 업데이트 완료 ({N}개 섹션).
이미지를 form_definitions/image/form_XX/ 에 보관하시겠습니까?
タイプ 분류 규칙이나 NET 계산식이 바뀌었다면 [config] 블록도 갱신하고 "form_XX 동기화해줘"로 재빌드해주세요.
```

텍스트 경로인 경우 (규칙 확정·업무규칙 변경):
```
form_XX.md 업데이트 완료 ({N}개 섹션).
실행 규칙이 바뀌었다면 [config] 블록을 함께 갱신하고 "form_XX 동기화해줘"로 form_types.json을 재빌드해 주세요.
```

단, ✅ 확정 항목이 변경된 경우에는 [config] 블록 갱신 + 재빌드를 자동으로 실행한다 — 사용자에게 물어보지 않는다:
```
form_XX.md 업데이트 완료 ({N}개 섹션).
확정된 규칙이 포함되어 있어 [config] 정본 블록을 갱신하고 form_types.json을 재빌드합니다.
```
→ 방금 수정한 form_XX의 `[config]` 블록을 바뀐 규칙에 맞게 갱신한 뒤 sync-form-config 스킬 Step 3(재빌드+검증)을 실행한다.

---

## 절대 규칙

- Step 4 포맷을 변형하지 않는다.
- Step 5 확인 없이 파일을 수정하지 않는다.
- 이미지에서 읽지 못한 내용을 추론으로 채우지 않는다. 불확실한 항목은 `(현업 확인 필요)`로 표기한다.
- 한 번에 하나의 form_XX.md만 수정한다.
